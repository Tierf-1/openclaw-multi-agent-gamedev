## OpenClaw 多智能体游戏开发团队系统开发方案

### 1. 目标与边界

本项目要在当前仓库已有的 `rules`、`agents`、`skills` 设定基础上，建设一套面向 **OpenClaw 游戏开发场景** 的多智能体团队系统。系统需要满足以下核心要求：

- **并行架构**：支持主从模式，允许多个执行型智能体并行完成模块级子任务。
- **上下文独立**：每个智能体拥有独立上下文存储，不共享对话历史，只通过结构化交付物协作。
- **沙盒隔离**：每个智能体只能写入自己的沙盒工作区，源规则目录只读。
- **规则复用**：尽量复用现有规则仓中的 Agent 分工、step 微流程、`skills` 按需加载模式。
- **CodeBuddy 验证**：在开发过程中，用当前仓库中的 `rules` 文档模拟 CodeBuddy 源文档，验证隔离与并行机制，但不得改写这些源文档。

### 2. 对现有设定的迁移策略

#### 2.1 可直接复用

- **角色分工**：制作人 / 项目管理 / 策划 / 主程 / 程序 / QA / UX / 美术。
- **规模模式**：XS/S/M 标准流程、L 主从流程、XL 深度流程。
- **微文件机制**：入口文件 + step 文件 + template 文件。
- **技能体系**：`rules/skills` 中的技能包作为可索引知识资产。
- **上下文恢复思想**：以结构化状态和交付物恢复运行状态。

#### 2.2 需要 OpenClaw 化改造

- 将 Unity 特定的运行目标抽象为 **OpenClaw 游戏项目工作流**。
- 将原先偏向 `.GameDev/` 的产物组织方式迁移为 **运行时目录 + 沙盒目录 + 交付目录**。
- 将“读取 step 文件执行”抽象为 **运行时 AgentSpec / SkillSpec / WorkflowSpec**。

### 3. 总体架构

### 3.1 架构分层

#### A. 规则资产层

负责扫描并索引：

- `rules/rule*.md`
- `rules/agents/*.md`
- `rules/agents/*/step-*.md`
- `rules/agents/*/templates/*.md`
- `rules/skills/**/*.md`

产出：

- 规则清单
- Agent 注册表
- Skill 注册表
- step/template 索引

#### B. 运行编排层

负责：

- 根据请求判断规模模式与流程模式
- 生成需求 ID、运行 ID、任务图
- 决定哪些阶段串行、哪些阶段并行
- 控制 handoff 与结果汇总

#### C. 上下文层

每个智能体单独维护：

- `context_id`
- 事件日志
- 输入摘要
- 输出摘要
- handoff 历史

上下文层设计要求：

- 各 Agent 日志文件分离
- 不允许直接读取其他 Agent 的上下文文件
- 只能通过 orchestrator 发布的交付物进行协作

#### D. 沙盒层

每个智能体运行时拥有独立目录：

- 独立工作目录
- 独立产物目录
- 独立上下文目录

沙盒层设计要求：

- 允许读取只读源目录
- 允许写入自己的 sandbox
- 禁止写入 `rules` 及其他受保护目录
- 通过显式路径校验防止越权写入

#### E. 验证层

针对 `codebuddy-validation` 目标：

- 视 `rules` 为 CodeBuddy 源规则文档
- 只读引用这些文档
- 在 `.openclaw-runtime/codebuddy-validation/` 中完成全部验证产物
- 额外执行“受保护目录写入尝试”，确保隔离机制会主动阻断

### 3.2 智能体拓扑

- **Producer**：需求识别、规模判断、流程路由
- **Project Manager**：状态登记、运行目录初始化、进度/恢复信息维护
- **Planner**：将需求转换为模块计划与交付清单
- **Lead Programmer**：形成技术拆分，生成并行子任务
- **Programmer x N**：并行实现模块级交付物
- **QA x N**：并行验证各模块交付物
- **Reporter**：汇总最终报告与验证结论

其中：

- `Producer / PM / Planner / Lead Programmer` 通常串行执行
- `Programmer x N` 和 `QA x N` 为并行阶段
- 所有执行型 Agent 都必须使用独立上下文和独立沙盒

### 4. 数据模型与交付协议

### 4.1 核心对象

- **AgentSpec**：从规则文档解析出的角色说明
- **SkillSpec**：从技能文档解析出的技能说明
- **RuleInventory**：当前规则仓的完整索引结果
- **RunContext**：一次运行的总上下文
- **AgentContextRecord**：某个 Agent 的独立上下文日志
- **SandboxPolicy**：读写白名单 / 黑名单规则
- **TaskPacket**：主 Agent 发给子 Agent 的结构化任务包
- **HandoffPacket**：子 Agent 返回的结构化交付物

### 4.2 交互协议

所有跨 Agent 协作统一走结构化包：

- 输入：`task.json`
- 输出：`handoff.json`
- 补充文档：`notes.md`

这样可以保证：

- 协作内容可追踪
- 不依赖共享上下文
- 更容易做恢复、审计与重放

### 6. 分阶段实施方案

### 阶段一：项目初始化

目标：建立项目可运行骨架。

交付：

- `pyproject.toml`
- CLI 入口
- 系统配置
- 验证目标配置
- README

### 阶段二：规则装载与注册

目标：让系统真正“基于现有 rules/agents/skills 设定”运行。

交付：

- 扫描规则目录
- 自动注册 Agent / Skill / step / template
- 输出 inventory 摘要

验收重点：

- 能识别全部 Agent 入口文件
- 能识别全部 Skill 文件
- 能建立 step/template 关联关系

### 阶段三：上下文与沙盒隔离

目标：落实“每个智能体独立上下文、独立沙盒”的硬要求。

交付：

- 每 Agent 独立 context 日志
- SandboxPolicy 路径校验
- 受保护目录写入阻断机制
- 运行时目录初始化逻辑

验收重点：

- Agent A 不能直接写 Agent B 的目录
- 任意 Agent 不能写 `rules`
- 所有可写产物都落在 `.openclaw-runtime/` 下

### 阶段四：并行编排

目标：实现主从并行执行。

交付：

- 需求路由器
- 主程拆分模块任务
- `asyncio` 并行执行 `Programmer x N`
- `asyncio` 并行执行 `QA x N`
- 统一汇总报告

验收重点：

- 并行阶段确实使用并发执行
- 每个并发 Agent 有独立 context / sandbox
- 汇总阶段只读取 handoff，不共享私有上下文

### 阶段五：CodeBuddy 隔离验证

目标：证明系统能在不污染源文档的前提下应用于 CodeBuddy 场景。

交付：

- `codebuddy-validation` 演示运行
- 读取 `rules` 作为源文档
- 在 `.openclaw-runtime/codebuddy-validation/` 生成验证产物
- 自动执行一次受保护路径写入拦截测试

验收重点：

- `rules` 中任意文件内容不被改写
- 报告明确记录并行执行结果
- 报告明确记录隔离校验结果

### 7. 风险与控制

- **风险：规则解析不稳定**  
  控制：优先解析标题、frontmatter、文件结构，不依赖复杂 Markdown 语义。

- **风险：并行执行变成伪并行**  
  控制：使用 `asyncio.gather`，并在测试中验证时序。

- **风险：沙盒隔离被路径穿透绕过**  
  控制：所有路径统一 `resolve()` 后再判断父级归属。

- **风险：CodeBuddy 验证误写源文档**  
  控制：把 `rules` 放进 `forbidden_write_roots`，并在演示中主动做拦截测试。

### 8. 测试策略

- **规则装载测试**：验证 Agent / Skill / step / template 数量与来源目录正确。
- **沙盒测试**：验证允许写自身目录、禁止写受保护目录。
- **并行测试**：验证多个子任务并发执行，且上下文/产物文件分离。
- **演示测试**：验证 `codebuddy-validation` 运行成功并产生报告。

### 9. 最终验收标准

满足以下条件即视为完成：

- 系统可扫描并注册现有 `rules` / `agents` / `skills`
- 系统具备主从并行执行能力
- 每个智能体具备独立上下文与独立沙盒
- CodeBuddy 验证流程在隔离目录中完成
- 受保护的源规则目录无法被写入
- 项目可通过 CLI 一键完成初始化、盘点与演示
